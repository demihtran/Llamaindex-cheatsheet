LI

Framework Overview

Data framework for LLM apps and RAG
Concept LlamaIndex connects private data to LLMs through loading, indexing, retrieval, and querying.
Use when RAG, chat over docs, structured extraction, agents use query engines/tools.
Install pip install -U llama-index or split provider packages.
Core Flow Load -> parse -> index -> retrieve -> synthesize -> evaluate.
pip install -U llama-index
pip install -U llama-index-llms-openai
pip install -U llama-index-embeddings-openai

from llama_index.core import (
    VectorStoreIndex,
    SimpleDirectoryReader,
    Settings,
)

documents = SimpleDirectoryReader("data").load_data()
index = VectorStoreIndex.from_documents(documents)
query_engine = index.as_query_engine()
response = query_engine.query("Summarize the docs")

Best: Start with a simple RAG pipeline, then optimize retrieval and synthesis layers separately.

Mistake: Treat LlamaIndex as only a vector DB wrapper; query engines and synthesis are also central.

SET

Settings

Global configuration for models and parsing
Concept Global settings replace old ServiceContext patterns for LLMs, embeddings, splitters, and callbacks.
Use when Set default model, embedding, and chunking configuration for the whole app.
Import from llama_index.core import Settings
Fields llm, embed_model, text_splitter, chunk_size.
from llama_index.core import Settings
from llama_index.llms.openai import OpenAI
from llama_index.embeddings.openai import OpenAIEmbedding
from llama_index.core.node_parser import SentenceSplitter

Settings.llm = OpenAI(model="gpt-4o-mini")
Settings.embed_model = OpenAIEmbedding(
    model="text-embedding-3-small"
)
Settings.text_splitter = SentenceSplitter(
    chunk_size=800,
    chunk_overlap=120,
)

Best: Configure model/embedding in one place, pin versions, and use env vars for keys.

Mistake: Mix embedding models in one index or use old ServiceContext patterns in new code.

LLM

LLMs & Embeddings

Model integrations for generation and vectors
Concept LLMs generate/synthesize; embedding models convert text into vectors.
Use when Query engine, response synthesis, agents, semantic retrieval.
Constructors OpenAI(model=...), OpenAIEmbedding(model=...)
Methods complete, chat, stream_complete, embedding qua index.
from llama_index.llms.openai import OpenAI
from llama_index.embeddings.openai import OpenAIEmbedding

llm = OpenAI(model="gpt-4o-mini", temperature=0.1)
embed_model = OpenAIEmbedding(model="text-embedding-3-small")

completion = llm.complete("Explain RAG in one sentence.")
chat = llm.chat([
    {"role": "system", "content": "Be concise."},
    {"role": "user", "content": "What is LlamaIndex?"},
])

Settings.llm = llm
Settings.embed_model = embed_model

Best: Choose the embedding model before building the index and keep it fixed for querying.

Mistake: Change the embedding model after building an index without re-indexing.

Loading Data

Readers, connectors, LlamaHub, LlamaParse
Concept Readers/loaders convert files, web pages, databases, and APIs into Document objects.
Use when Ingest data before when parse/chunk/index.
Import SimpleDirectoryReader and connector packages from LlamaHub.
Methods load_data(), connector-specific options.
from llama_index.core import SimpleDirectoryReader

documents = SimpleDirectoryReader(
    input_dir="data",
    recursive=True,
    required_exts=[".pdf", ".md", ".txt"],
).load_data()

for doc in documents:
    print(doc.id_, doc.metadata)

# LlamaParse for complex PDFs/tables:
# pip install llama-parse
# from llama_parse import LlamaParse

Best: Keep metadata such as source, page, and tenant_id for filtering, citation, and debugging.

DOC

Documents & Nodes

Core data units before and after parsing
Document Raw/semi-raw text + metadata from loaders.
Node Chunk with ID, text, metadata, relationships, and optional embedding.
Import Document, TextNode
Methods get_content(), set_content(), metadata fields.
from llama_index.core import Document
from llama_index.core.schema import TextNode

doc = Document(
    text="LlamaIndex builds RAG pipelines.",
    metadata={"source": "notes.md", "section": "intro"},
)

node = TextNode(
    text="A node is a chunk for retrieval.",
    metadata={"source": "notes.md"},
)

print(doc.text)
print(node.get_content())

Best: Use Document for ingestion and Node for indexing/retrieval; keep metadata consistent.

Mistake: Create chunks without source/page metadata so responses cannot cite evidence.

NP

Node Parsers

Split documents into retrieval-ready nodes
Concept Node parser/chunker create nodes from documents.
Use when Long PDFs/docs, code, Markdown, and corpora that need retrieval.
Constructor SentenceSplitter(chunk_size, chunk_overlap)
Methods get_nodes_from_documents(), transformations in ingestion.
from llama_index.core.node_parser import SentenceSplitter

splitter = SentenceSplitter(
    chunk_size=800,
    chunk_overlap=120,
)

nodes = splitter.get_nodes_from_documents(documents)

index = VectorStoreIndex(nodes)

Best: Tune chunking with real queries and evaluate retrieval recall.

Mistake: Chunks that are too small lose context; chunks that are too large reduce recall.

ING

Ingestion Pipeline

Transform, parse, extract metadata, embed, cache
Concept Pipeline runs transformations from documents into index-ready nodes.
Use when Production ingestion, caching, metadata extraction, batch processing.
Import IngestionPipeline, SentenceSplitter, embedding transforms.
Methods run(), arun(), optional cache/vector_store.
from llama_index.core.ingestion import IngestionPipeline
from llama_index.core.node_parser import SentenceSplitter
from llama_index.embeddings.openai import OpenAIEmbedding

pipeline = IngestionPipeline(
    transformations=[
        SentenceSplitter(chunk_size=800, chunk_overlap=120),
        OpenAIEmbedding(model="text-embedding-3-small"),
    ],
)

nodes = pipeline.run(documents=documents)

# You can add metadata extractors, custom transforms,
# cache, or a vector_store for production ingestion.

Best: Build ingestion outside the query path and cache expensive transformations.

Mistake: Re-embed the whole corpus on every app start or skip pipeline versioning.

IDX

VectorStoreIndex

Most common index for semantic search and RAG
Concept Index embeds nodes/documents and store into vector store.
Use when RAG over documents, semantic search, and chat over private data.
Constructors from_documents(), VectorStoreIndex(nodes)
Methods as_query_engine, as_retriever, as_chat_engine, insert.
from llama_index.core import VectorStoreIndex

index = VectorStoreIndex.from_documents(
    documents,
    show_progress=True,
)

query_engine = index.as_query_engine(
    similarity_top_k=4,
)
response = query_engine.query("What are the main findings?")

retriever = index.as_retriever(similarity_top_k=5)
nodes = retriever.retrieve("main findings")

Best: Set similarity_top_k by evaluation, not by intuition alone.

Mistake: Rebuild indexes in the request path or fail to persist the index/vector store.

STO

StorageContext

Persist index, docstore, index store, vector store
Concept StorageContext manages where nodes, index metadata, and vector data are stored.
Use when Persist/load index, custom vector DB, production RAG.
Import StorageContext, load_index_from_storage
Methods persist(), from_defaults().
from llama_index.core import (
    StorageContext,
    load_index_from_storage,
)

# Persist
index.storage_context.persist(persist_dir="./storage")

# Load later
storage_context = StorageContext.from_defaults(
    persist_dir="./storage"
)
index = load_index_from_storage(storage_context)

Best: Use a durable vector store and backed-up storage in production.

Mistake: Only use temporary local storage or load an index with a mismatched embedding model.

VS

Vector Stores

External databases for embeddings and metadata filters
Concept Backend store vectors/metadata: Chroma, Qdrant, Pinecone, Weaviate, Postgres...
Use when Large corpora, persistence, filtering, hybrid search, and production scale.
Import Provider-specific vector store + StorageContext.
Methods Use qua VectorStoreIndex.from_vector_store() or storage context.
# Example shape; install provider package first.
from llama_index.core import StorageContext, VectorStoreIndex
from llama_index.vector_stores.chroma import ChromaVectorStore

vector_store = ChromaVectorStore(chroma_collection=collection)
storage_context = StorageContext.from_defaults(
    vector_store=vector_store
)

index = VectorStoreIndex.from_documents(
    documents,
    storage_context=storage_context,
)

Best: Choose stores with filtering, update/delete paths, backups, and tenancy support for your data.

Mistake: Forget metadata filters for access control or multi-tenant data.

QE

Query Engine

Question answering over indexes with retrieval + synthesis
Concept Query engines accept queries, retrieve nodes, and synthesize responses.
Use when Simple RAG Q&A APIs, document Q&A, and one-shot answers.
Constructor index.as_query_engine(...), RetrieverQueryEngine.from_args(...)
Methods query(), aquery(), source nodes in response.
query_engine = index.as_query_engine(
    similarity_top_k=4,
    response_mode="compact",
)

response = query_engine.query(
    "What does the handbook say about refunds?"
)

print(response)

for source in response.source_nodes:
    print(source.score, source.node.metadata.get("source"))

# Async
# response = await query_engine.aquery("...")

Best: expose source nodes/citations; tune top_k + response_mode qua eval.

Mistake: Only read the final answer instead of inspecting retrieved nodes when answers are wrong.

RT

Retrievers

Fetch relevant nodes before synthesis
Concept Retrievers return relevant nodes but do not generate answers.
Use when Custom RAG, rerank, inspect recall, compose query engines.
Constructor index.as_retriever(similarity_top_k=...)
Methods retrieve(), aretrieve().
retriever = index.as_retriever(
    similarity_top_k=8,
)

nodes = retriever.retrieve("refund policy")

for n in nodes:
    print(n.score)
    print(n.node.get_content()[:300])

# Combine with rerank/postprocessors
# before response synthesis.

Best: test retriever separate with labelled queries before when blame LLM.

Mistake: raise top_k too cao make synthesis gets noisy and waste token.

RS

Response Synthesizer

Turn retrieved nodes into final answers
Concept Module that synthesizes answers from the query plus retrieved nodes.
Use when Custom retrieval pipeline, thay response mode, citation strategy.
Constructor get_response_synthesizer(response_mode=...)
Modes compact, refine, tree_summarize, no_text.
from llama_index.core import get_response_synthesizer
from llama_index.core.query_engine import RetrieverQueryEngine

synthesizer = get_response_synthesizer(
    response_mode="compact",
)

query_engine = RetrieverQueryEngine.from_args(
    retriever=retriever,
    response_synthesizer=synthesizer,
)

response = query_engine.query("Summarize the contract")

Best: Choose response mode by document/task type: short QA, long summaries, or citations.

Mistake: Use one response mode for every use case without measuring quality/cost.

PP

Node Postprocessors

Rerank, filter, trim, reorder retrieved nodes
Concept Postprocessors modify retrieved nodes before synthesis.
Use when Reranking, metadata filtering, similarity cutoff, context compression.
Import SimilarityPostprocessor, reranker packages.
Parameter node_postprocessors=[...] in query engine.
from llama_index.core.postprocessor import SimilarityPostprocessor

query_engine = index.as_query_engine(
    similarity_top_k=10,
    node_postprocessors=[
        SimilarityPostprocessor(similarity_cutoff=0.72),
    ],
)

response = query_engine.query("What changed in v2?")

Best: Rerank/cutoff reduces context noise for large corpora.

Mistake: cutoff too cao make lose recall or not log nodes gets removed.

CE

Chat Engine

Conversational querying over data
Concept Chat engines keep conversation context and query the index turn by turn.
Use when Chatbot over docs, support assistant, conversational RAG.
Constructor index.as_chat_engine(chat_mode=...)
Methods chat(), stream_chat(), reset().
chat_engine = index.as_chat_engine(
    chat_mode="context",
    similarity_top_k=4,
    system_prompt="Answer from the docs. Cite sources.",
)

response = chat_engine.chat("What is our refund policy?")
print(response)

stream = chat_engine.stream_chat("Summarize it in bullets")
for token in stream.response_gen:
    print(token, end="")

Best: Use chat engines for multi-turn use cases, while still limiting history/context.

Mistake: Use a stateless query engine for chatbots and manually concatenate history.

TL

Tools

Functions and query engines exposed to agents
Concept Tools are callables/query engines with metadata that agents use for selection.
Use when Agents need calculators, APIs, search, or specialized query engines.
Import FunctionTool, QueryEngineTool, ToolMetadata
Methods from_defaults(); callable functions should have type hints and docstrings.
from llama_index.core.tools import (
    FunctionTool,
    QueryEngineTool,
    ToolMetadata,
)

def multiply(a: float, b: float) -> float:
    """Multiply two numbers."""
    return a * b

math_tool = FunctionTool.from_defaults(fn=multiply)

docs_tool = QueryEngineTool(
    query_engine=query_engine,
    metadata=ToolMetadata(
        name="docs_search",
        description="Search internal docs for policy answers.",
    ),
)

Best: Use specific tool names/descriptions; functions should have clear type hints and docstrings.

Mistake: agent has too many tools ambiguous or tool side effect not guardrail.

AG

Agents

FunctionAgent, AgentWorkflow, ReActAgent, CodeActAgent
Concept Agents use the LLM to choose tools across multiple steps until the task is complete.
Use when The model must decide which query engine, tool, or API to call.
Import from llama_index.core.agent.workflow import FunctionAgent
Methods Async run(user_msg=...), with streaming depending on the model/runtime.
from llama_index.core.agent.workflow import FunctionAgent
from llama_index.llms.openai import OpenAI

def add(a: float, b: float) -> float:
    """Add two numbers."""
    return a + b

agent = FunctionAgent(
    tools=[add, docs_tool],
    llm=OpenAI(model="gpt-4o-mini"),
    system_prompt=(
        "You answer questions and use tools when needed."
    ),
)

response = await agent.run(
    user_msg="What is 20 + 22? Then check docs for refund rules."
)
print(response)

Best: Use agents for tool selection; use query engines for fixed QA workflows.

Mistake: Let agents decide everything when the retrieval pipeline should be deterministic.

SO

Structured Output

Pydantic models and structured prediction
Concept Constrain output with a Pydantic schema instead of parsing free-form text.
Use when Extraction, classification, agent final answer, typed API output.
Import BaseModel; LLM structured predict APIs.
Methods structured_predict(), agent structured output patterns.
from pydantic import BaseModel, Field
from llama_index.llms.openai import OpenAI

class ContactInfo(BaseModel):
    name: str = Field(description="Person name")
    email: str
    company: str | None = None

llm = OpenAI(model="gpt-4o-mini")

info = llm.structured_predict(
    ContactInfo,
    "Extract contact info: Lan, lan@example.com",
)

print(info.email)

Best: Use strict schemas, short field descriptions, and edge-case validation.

Mistake: Ask for JSON in a prompt without validating against a schema.

WF

Workflows

Event-driven, step-based orchestration for complex apps
Concept Workflows split an app into steps that receive Events and return the next Event.
Use when Branches, loops, async orchestration, human-in-the-loop, agentic RAG.
Import Workflow, step, Event, StartEvent, StopEvent
Methods run(), handler awaitable, stream_events().
from llama_index.core.workflow import (
    Workflow,
    step,
    Event,
    StartEvent,
    StopEvent,
)

class Retrieved(Event):
    context: str

class RAGWorkflow(Workflow):
    @step
    async def retrieve(self, ev: StartEvent) -> Retrieved:
        nodes = retriever.retrieve(ev.question)
        context = "\n\n".join(n.node.get_content() for n in nodes)
        return Retrieved(context=context)

    @step
    async def answer(self, ev: Retrieved) -> StopEvent:
        response = llm.complete(f"Answer from context:\n{ev.context}")
        return StopEvent(result=str(response))

result = await RAGWorkflow(timeout=60).run(question="What changed?")

Best: type events clear; use plain Python for branches/loops; validate workflow in tests.

Mistake: Hide complex workflow logic inside query-engine callbacks that are hard to debug.

PR

Prompts

Customize templates used by query engines and LLMs
Concept Prompt templates control synthesis, QA, refine steps, agents, and structured extraction.
Use when Domain-specific tone, citation rules, extraction instructions.
Import PromptTemplate
Methods get_prompts(), update_prompts(), template formatting.
from llama_index.core import PromptTemplate

qa_prompt = PromptTemplate(
    "Context:\n{context_str}\n\n"
    "Question: {query_str}\n"
    "Answer in Vietnamese and cite sources."
)

query_engine.update_prompts({
    "response_synthesizer:text_qa_template": qa_prompt,
})

prompts = query_engine.get_prompts()

Best: Keep prompts short, test them with an eval set, and version prompt changes.

Mistake: Try to fix hallucination with prompts while retrieval context is wrong or missing.

RO

Routers

Route queries to the best tool or query engine
Concept Routers choose the appropriate query engine/tool based on query and metadata.
Use when Multiple indexes by domain, summary vs vector search, SQL vs docs.
Import Router query engines/selectors from llama_index.core.
Pattern Build tools -> selector -> router engine.
from llama_index.core.tools import QueryEngineTool, ToolMetadata
from llama_index.core.query_engine import RouterQueryEngine
from llama_index.core.selectors import LLMSingleSelector

policy_tool = QueryEngineTool(
    query_engine=policy_engine,
    metadata=ToolMetadata(
        name="policy_docs",
        description="Use for HR and refund policy questions.",
    ),
)

router = RouterQueryEngine(
    selector=LLMSingleSelector.from_defaults(),
    query_engine_tools=[policy_tool, api_tool],
)

Best: Tool descriptions must clearly distinguish domain and data scope.

Mistake: Stack too many routers without tracing selection decisions.

PG

Property Graph Index

Graph-based retrieval over entities and relationships
Concept Index extracts entities/relations into a graph for relationship-oriented queries.
Use when Knowledge graph, entity-centric QA, relationship-heavy corpora.
Import PropertyGraphIndex
Methods from_documents(), as_query_engine(), graph store integrations.
from llama_index.core import PropertyGraphIndex

graph_index = PropertyGraphIndex.from_documents(
    documents,
    show_progress=True,
)

query_engine = graph_index.as_query_engine(
    include_text=True,
)

response = query_engine.query(

)

Best: Use when relationships/entities matter more than pure similarity search.

Mistake: Use a graph index for a simple corpus and make the pipeline slower and more complex.

EV

Evaluation

Measure retrieval and response quality
Concept Evaluators check faithfulness, relevancy, correctness, and retrieval quality.
Use when Tuning chunking/top_k/prompts, regression tests for RAG.
Import Evaluators from llama_index.core.evaluation.
Pattern Dataset queries -> run pipeline -> evaluator -> compare versions.
from llama_index.core.evaluation import (
    FaithfulnessEvaluator,
    RelevancyEvaluator,
)

faithfulness = FaithfulnessEvaluator(llm=llm)
relevancy = RelevancyEvaluator(llm=llm)

response = query_engine.query("What is the SLA?")

faith_result = faithfulness.evaluate_response(
    response=response,
)
rel_result = relevancy.evaluate_response(
    query="What is the SLA?",
    response=response,
)

Best: Create a small but realistic eval set; track retrieval and generation separately.

Mistake: Tune with a few manual questions and deploy without regression tests.

OBS

Observability

Callbacks, tracing, debugging, token counting
Concept Callback/instrumentation trace retrieval, LLM calls, token usage, latency.
Use when Debug wrong answers, cost tracking, production monitoring.
Import CallbackManager, debug/token handlers, external tracing tools.
Config Set through Settings.callback_manager.
from llama_index.core import Settings
from llama_index.core.callbacks import (
    CallbackManager,
    LlamaDebugHandler,
)

debug = LlamaDebugHandler(print_trace_on_end=True)
Settings.callback_manager = CallbackManager([debug])

response = query_engine.query("Why did retrieval fail?")

Best: log retrieved nodes, prompts, latency, token usage, model/index versions.

Mistake: Only log the final answer; without traces it is hard to know whether retrieval or synthesis failed.

PRO

Production RAG

Operational checklist for reliable LlamaIndex apps
Ingestion Incremental, idempotent, versioned transforms, metadata filters.
Query Trace retrieval, postprocess/rerank, return citations.
Storage Durable vector store, backups, delete/update path.
Safety PII handling, tenant filters, eval gates, cost controls.
# Production pattern
1. Load docs with stable IDs and metadata.
2. Transform/chunk/embed in an ingestion job.
3. Persist to vector store + document store.
4. Query via retriever + postprocessors.
5. Synthesize with citations and source nodes.
6. Evaluate retrieval and answer quality.
7. Trace every run and monitor cost/latency.

Best: separate ingestion service from query API; version everything.

Mistake: doing data ingestion, index build, and user query in one request path.

AR

Agentic RAG Pattern

Multiple query engines as tools, chosen by an agent
from llama_index.core.tools import QueryEngineTool, ToolMetadata
from llama_index.core.agent.workflow import FunctionAgent

policy_tool = QueryEngineTool(
    query_engine=policy_index.as_query_engine(similarity_top_k=5),
    metadata=ToolMetadata(
        name="policy_docs",
        description="Use for HR, security, and compliance policy questions.",
    ),
)

api_tool = QueryEngineTool(
    query_engine=api_index.as_query_engine(similarity_top_k=8),
    metadata=ToolMetadata(
        name="api_docs",
        description="Use for SDK, API endpoint, error code, and integration questions.",
    ),
)

agent = FunctionAgent(
    tools=[policy_tool, api_tool],
    llm=llm,
    system_prompt=(
        "Choose the most relevant tool. Cite source nodes. "
        "If neither tool has evidence, say you do not know."
    ),
)

response = await agent.run(
    user_msg="What does error E_AUTH_42 mean and who owns the fix?"
)

Use this for: multiple knowledge bases, mixed APIs/docs, domain-specific tools, and questions requiring tool selection.

Avoid it for: simple single-index Q&A where a deterministic query engine is easier to evaluate and debug.

RET

Retrieval Tuning

Improve recall and precision before blaming the LLM
  • similarity_top_k
    Start small, evaluate recall, then increase only when evidence is missing.
  • Node postprocessors
    Similarity cutoff, metadata filters, recency filters, rerankers, context trimming.
  • Hybrid search
    Use when exact terms, error codes, names, or IDs matter as much as semantics.
  • RouterQueryEngine
    Route between vector search, summary index, SQL, graph, or external tools.
  • PropertyGraphIndex
    Use when entity relationships are the real retrieval target.
  • Eval loop
    Track retrieved nodes, response text, source accuracy, cost, latency, and version ids.
PRO

Real-Life LlamaIndex Architecture

Production RAG is a data lifecycle, not only a query engine
# Production RAG lifecycle
source_systems
  -> loader / reader
  -> document normalization
  -> node parser + metadata extraction
  -> embeddings + vector store write
  -> retriever + postprocessors + synthesizer
  -> answer with source nodes
  -> eval + trace + feedback

# Version all of these:
data_version = "docs_2026_06_30"
pipeline_version = "splitter_512_overlap_80_v3"
embedding_version = "text-embedding-3-large"
prompt_version = "qa_citation_v5"
  • Separate ingestion
    Do not parse, chunk, embed, and query inside one user request path.
  • Stable metadata
    Every node needs source, page/section, tenant, permission, timestamp, and document version.
  • Persist everything
    Use persistent vector store, docstore/index store when needed, and backup storage.
  • Inspect source_nodes
    When answers fail, inspect retrieved nodes before changing prompts or models.
  • Postprocess
    Use similarity cutoff, reranking, metadata filters, recency filters, and context trimming.
  • Measure versions
    Compare retrieval quality whenever chunking, embedding, top_k, or response mode changes.
IDX

Index Maintenance Best Practices

Keep the knowledge base fresh, correct, and debuggable
  • Stable IDs
    Derive document and node ids from source path/url, section, version, and content hash.
  • Incremental updates
    Upsert changed documents, delete removed documents, and tombstone stale nodes.
  • Embedding migration
    Changing embedding model means building a new index; do not mix vector spaces.
  • Access control
    Filter by tenant/user permissions before synthesis, not after the final answer.
  • Quality dashboards
    Track empty retrieval, low-score retrieval, no-source answers, stale source usage, and cost.
  • Rollback
    Keep previous index versions so bad ingestion jobs can be rolled back quickly.
OBS

Eval & Observability

Separate retrieval failures from synthesis failures
  • Retrieval eval
    Measure whether expected nodes appear in top-k, before asking if the final answer is good.
  • Faithfulness
    Check whether claims are supported by retrieved source nodes.
  • Relevancy
    Check whether the answer addresses the query without drifting.
  • Source accuracy
    Citations must support the exact claim, not merely come from a related document.
  • Trace fields
    Log query, retrieved node ids, scores, postprocessor decisions, response mode, model, latency, tokens.
  • Feedback loop
    Convert bad user sessions into labelled eval cases before changing prompts blindly.

Quick Decision Map
